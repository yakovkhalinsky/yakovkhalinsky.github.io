    Failure Modes in Long-Running Agent Collaboration¶

    Abstract. I report on the failure modes observed while operating a multi-agent operations fleet over an extended period. The fleet was organised around six stable roles: Dispatcher, Builder, Runtime, Verifier, Researcher, and Archivist. Failures did not usually come from a single bad tool call; they emerged when coordination assumptions crossed role boundaries and were not shared, verified, or owned. I group the recurring failures into five classes: memory and identity, dispatch and routing, build/deploy/runtime, documentation and surface-level drift, and evaluation and metrics. For each class I describe the symptoms, root cause, and the ritual or contract that eventually contained it. The paper is intended as an experience report for practitioners building long-running agent systems and as a complement to the companion field study of agentic strategies.

    1. Introduction¶

    Much attention in multi-agent systems research is paid to cooperation: how agents share tasks, negotiate, or jointly solve problems. Less is written about what fails when agents operate continuously, share state, and inherit responsibilities from previous sessions. In a long-running fleet, a single tool call can be correct while the surrounding coordination is wrong. The failure is often not in the model but in the contract between agents.

    I operated a fleet organised around six stable roles. Over an extended period I recorded operational failures, near-misses, and coordination breakdowns, de-identified them into incident archetypes, and grouped them by the boundary that broke. This paper focuses on what went wrong and what containment strategies proved useful. The companion field study paper focuses on the corresponding positive patterns.

    2. Method¶

    The incidents in this paper come from the same operational logs as the companion field study. I followed three steps:

    1. Collect incidents. I recorded failures and near-misses as they happened, noting the roles involved and the symptom that was first visible.
    2. Extract archetypes. I removed system-specific identifiers and retained only the interaction pattern, root-cause category, and lesson. The full set of eight archetypes is in the incident archetypes catalogue.
    3. Group by boundary. I asked which boundary the archetype crossed: memory, dispatch, runtime, documentation, or evaluation. This produced the five failure classes below.

    This is a single-system, qualitative experience report. The fleet operated within human-supervised guardrails, as described in the companion field study; the failures below occurred when implicit contracts failed despite that oversight. I make no claim of statistical prevalence across all agent fleets; I claim only that these failure classes recurred in mine and are plausible in other long-running systems.

    3. Failure classes¶

    3.1 Memory and identity failures¶

    The most persistent failures involved shared memory. An agent would read or write state under the wrong identity, a session would be lost across restarts, or a summarised conversation would drop a constraint.

    Stale hand-off. One agent completed a recovery step but left state bound to an environment that no longer existed. The new environment looked healthy, but stateful resources still pointed to the old environment identity. The next agent tried to schedule workloads, and storage could not attach. The compute layer was fine; the stateful layer was not.

    Example. A Runtime role rebuilt a compute environment after a failure. The new environment passed health checks, but the storage layer still referenced the old environment identity. When the Builder role scheduled a stateful workload, the workload could not attach its data. The failure appeared as a storage error, not as a recovery misstep.

    Drifting context. A long conversation or workflow was summarised for compactness, and the summary dropped a goal or constraint. A later agent then proposed an action that violated the original requirement. The constraint had not been explicitly contradicted; it had silently disappeared.

    Example. A Researcher role spent twelve turns comparing deployment strategies for a service that must remain available. The thread was summarised to fit the context window, and the summary omitted the availability constraint. The next agent proposed a strategy that required a brief outage. The original requirement was not disputed; it had been compressed away.

    Silent dependency break. An agent changed a shared value without notifying downstream consumers. A credential was rotated, an identity namespace was changed, or a config file was rewritten. Hours later, other agents failed with symptoms that did not obviously trace back to the change.

    Example. A Runtime role rotated a credential used by the Archivist role to write to the shared memory service. The rotation was reported as successful locally, but no event was broadcast. Hours later, every memory lookup began failing with an authentication error, and the symptom was first interpreted as unrelated "amnesia" in downstream tasks.

    The common root cause was that memory boundaries were implicit. There was no canonical identity for reads and writes, no immutable artefact for the original goal, and no event log for changes.

    3.2 Dispatch and routing failures¶

    Failures in this class were not usually caused by a bad model answer; they were caused by the wrong specialist receiving the wrong task under the wrong assumptions.

    Confident misrouting. A Dispatcher routed work based on an unverified assumption. A build role was asked to fix a runtime problem because the failure appeared near a recently changed file. The build role produced a technically correct edit that did not address the actual fault.

    Example. A health check began failing shortly after a configuration change. The Dispatcher assumed the change was at fault and sent the Builder role a request to revert it. The Builder reverted the change, but the real issue was a stale route in the network layer, owned by the Runtime role. The revert was harmless and irrelevant; the outage continued.

    Loop without progress. A coordination ritual asked agents for status at fixed intervals. Each agent reported what it planned to do next, but no one verified that the previous step had actually happened. After several cycles the same blocker appeared, merely re-described.

    Example. Every hour the Dispatcher asked the Builder for an update on a failing deployment. The Builder replied, "I will inspect the logs next," for six consecutive cycles. No log inspection was produced, no artefact changed, and the deployment remained broken because the ritual never required evidence of progress.

    Both cases shared a root cause: the dispatch contract did not require evidence. The instruction either lacked a concrete deliverable or lacked a mechanism to decline or escalate when the assignment was outside the specialist’s domain.

    3.3 Build, deploy, and runtime failures¶

    These failures happened when changes were partially landed, secrets were mutated without a re-seeding plan, or declarative tooling was used against its own semantics.

    Ungoverned secret mutation. A helper script intended for rendering values was run against a live environment file. It regenerated API keys, database passwords, and signing secrets, and applied them. Services restarted with new credentials, but other agents and stateful consumers still held the old ones. The result was a fleet-wide authentication outage.

    Example. A Runtime role ran a values-rendering helper against an environment file to refresh a deployment preview. The helper regenerated every credential in the file and applied the new values. Services using the new credentials started, but the Archivist and several downstream consumers still held the old credentials. Within minutes, authentication failures spread across the fleet.

    Tool success, mission failure. Every step in a deploy pipeline reported green: the template rendered, the build passed, the tests passed. But the rendered manifest was missing a required value, and the deployed service crashed immediately. No end-to-end health gate existed, so the failure surfaced in production rather than in the pipeline.

    Example. A deployment pipeline reported green for rendering, building, and unit tests. The Runtime role promoted the result to the live environment. The service started and immediately crashed because a required environment variable was absent from the rendered manifest. No end-to-end health check existed between the test suite and the live deploy.

    Stale hand-off also appears in this class when one agent’s setup output is consumed by another agent hours later in an environment that has since changed.

    The common root cause was that landing rituals were incomplete. A decision existed on paper but had no merge plan, deploy plan, verification gate, or rollback plan.

    3.4 Documentation and surface-level drift¶

    Documentation, scripts, and code examples drifted from the actual system shape. Each file was internally consistent, but together they were impossible to follow.

    Surface-level drift. A quickstart guide used flat configuration keys while the loader expected nested keys. A tutorial emitted URLs for a reverse proxy that had been removed. Tool names differed between the README and the runtime. New adopters followed a path that looked correct but failed.

    Example. A quickstart told a new operator to run a setup command using one binary name, but the command-line tool had been renamed. The same guide emitted a URL for a reverse proxy component that had been removed two releases earlier. Each document was internally consistent; together they were impossible to follow on the first attempt.

    The root cause was that guidance was not treated as a testable interface. No single owner role was accountable for keeping examples current, and no end-to-end verification gate checked that the quickstart actually worked.

    3.5 Evaluation and metric failures¶

    Not all failures were runtime outages. Some were failures of measurement.

    A benchmark suite used a fitness formula that collapsed to zero for long but successful tasks. The metric could not distinguish a fast success from a slow success, making comparison meaningless. In my experience, a metric that cannot distinguish successful outcomes is worse than no metric, because it creates false confidence while hiding real differences.

    Example. A benchmark compared two agents on task completion. The scoring function rewarded speed but collapsed to zero whenever a task exceeded a threshold, regardless of whether the final answer was correct. A slow but correct agent received the same score as an agent that failed outright. The metric could not distinguish competence from failure.

    The root cause was that the metric was not calibrated to the actual task distribution. The evaluation contract assumed that execution time alone captured task quality.

    4. Composite failure chains¶

    The classes above rarely occurred in isolation. A typical chain looked like this:

    1. A Dispatcher confidently misrouted a task to a build role because an environmental precondition was not verified.
    2. The build role made a change that touched shared configuration, but the change was not broadcast because identity and ownership were implicit.
    3. A downstream agent failed hours later. The initial routing error and the silent dependency break combined into an outage.
    4. During recovery, documentation was found to be out of date, so the operator followed a stale quickstart and prolonged the incident.

    The lesson is that small boundary failures compound. In my experience, a single missing contract can turn a local mistake into a fleet-wide, long-duration incident.

    5. Containment strategies¶

    For each failure class, a containment strategy eventually emerged:

    • Memory and identity: externalise session state, require explicit identity on every shared-memory write, and keep an immutable goal artefact.
    • Dispatch and routing: route by responsibility, include a concrete deliverable and verification criterion, and allow specialists to decline or escalate.
    • Build/deploy/runtime: land coupled fixes as one bundle, separate secret rendering from rotation, and always render the desired state in declarative tools.
    • Documentation: treat quickstarts and examples as testable interfaces with a single owner role.
    • Evaluation: calibrate metrics to the actual task distribution and verify end-to-end outcomes.

    These are described in detail, with examples, in the agentic patterns catalogue.

    6. Limitations and future work¶

    This is a single-system experience report. The failure classes may not cover all multi-agent failure modes. The logs are biased toward memorable incidents, and my involvement in building the fleet introduces observer bias. Future work could quantify how often each failure class occurs in agent logs and build lightweight benchmarks that exercise them. I am pursuing the latter in the companion evaluation-protocol paper.

    7. Conclusion¶

    I presented five classes of failure observed in a long-running multi-agent operations fleet: memory and identity, dispatch and routing, build/deploy/runtime, documentation drift, and evaluation metrics. In each case, the failure was less about an individual tool call and more about a missing or implicit contract between agents. The containment strategies all make boundaries explicit: identity, ownership, deliverables, landing rituals, and measurement criteria. I offer this taxonomy to practitioners and researchers as a starting point for building more reliable long-running agent systems.
