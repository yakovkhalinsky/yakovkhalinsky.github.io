    Agentic Strategies and Patterns catalogue¶

    A catalogue of recurring strategies, good patterns, and anti-patterns observed in a long-running multi-agent operations fleet. Each entry is derived from anonymised incident archetypes and operational experience.

    The catalogue is organised by the concern it addresses: dispatch, memory, verification, build/deploy, and coordination rituals. It is intended to feed the paper set and the role taxonomy.

    1. Dispatch patterns¶

    Pattern 1.1: Role-based routing¶

    Statement: Assign work to agents by responsibility, not by availability or message proximity.

    How it works: A Dispatcher reads the task domain and routes to the agent whose role owns that domain: Builder for code, Runtime for infrastructure, Verifier for correctness gates, Researcher for options, Archivist for memory and guidance.

    Why it helps: Clear ownership prevents multiple agents from making partial edits to the same artefact. It also makes misrouting auditable.

    When it fails: When the Dispatcher routes based on keyword matching without confidence (see anti-pattern 1.2), or when boundaries between roles are vague.

    Linked archetype: confident-misrouting

    Example. A user asks the fleet to "fix the slow endpoint." The Dispatcher routes the task to the Runtime role for infrastructure diagnosis, rather than to the Builder, because the symptom points to a service health problem. The Builder is only pulled in once the Runtime role confirms the issue is in code.

    Pattern 1.2: Verifiable dispatch instruction¶

    Statement: Every dispatch must include a concrete deliverable and a verification criterion.

    How it works: Instead of asking an agent to “look at X,” the Dispatcher says: “Fix X so that test Y passes; do not commit until Z is green.”

    Why it helps: Status updates can be faked or stalled; a verifiable deliverable forces closure.

    Linked archetype: loop-without-progress

    Example. The Dispatcher sends the Builder: "Update the health-check endpoint so that the fleet-wide liveness probe returns HTTP 200 within 2 seconds; do not mark the task complete until the Verifier reports the probe green on the staging fleet." The Builder cannot reply with a status update alone; the green/red verdict from the Verifier is the required deliverable.

    Anti-pattern 1.1: Status-only check-ins¶

    Statement: A coordination ritual that asks agents for status but not for a deliverable.

    Symptoms: Agents report next steps repeatedly without completing them; the same blocker appears in every report.

    Consequence: The fleet keeps moving without making progress.

    Linked archetype: loop-without-progress

    Example. Every hour the Dispatcher asks the Builder for an update on a failing deploy. The Builder replies, "I will check the logs next," for six cycles. No log check happens, no artefact is produced, and the deploy remains broken because the ritual never required evidence of progress.

    Anti-pattern 1.2: Confident misrouting¶

    Statement: A Dispatcher assigns work to a specialist based on assumptions that are not verified.

    Symptoms: A build agent is asked to fix a runtime problem; a runtime agent is asked to edit code. The specialist proceeds anyway because the instruction does not allow declining.

    Consequence: Subtle wrong outputs, wasted time, and blame diffusion.

    Linked archetype: confident-misrouting

    Example. The Dispatcher sees a runtime alert and assumes it is caused by a recent configuration change. It sends the Builder a request to revert the change. The Builder complies, but the real fault is a stale network rule managed by the Runtime role. The revert is harmless and irrelevant; the outage continues.

    2. Memory and shared-state patterns¶

    Pattern 2.1: Externalise session state¶

    Statement: Session affinity must survive replica restart and horizontal scaling.

    How it works: Store session metadata in a shared service (e.g., Redis) with an in-memory fallback for single-process or degraded modes.

    Why it helps: Agents can scale without losing conversation context; recovery becomes stateless.

    Linked archetype: drifting-context

    Example. A conversation spans ten turns. A deployment restarts the agent process mid-task. Because session metadata is held in a shared memory service, the restarted agent resumes the same conversation state. Without externalisation, the agent defaults to a generic identity and asks the user to restate the goal.

    Pattern 2.2: Explicit identity in shared tools¶

    Statement: Tools that write shared memory must require the caller’s identity and the subject’s identity; defaults must not encode a single-agent assumption.

    How it works: A memory tool rejects a write unless both the caller identity and the subject identity are supplied.

    Why it helps: Prevents silent misrouting into the wrong memory bucket in multi-agent systems.

    Linked archetype: silent-dependency-break

    Example. The Archivist role maintains a memory tool. Early on, the tool defaulted to a generic bucket when the caller identity was omitted. One agent wrote critical runtime notes into that bucket, but another agent later queried a different bucket and found nothing. Requiring explicit identity fields on every write turned the hidden bug into a visible contract violation.

    Pattern 2.3: Local-first, shared-when-useful¶

    Statement: Keep transient, per-agent state in local memory; promote only stable, cross-agent facts to shared memory.

    How it works: Agents tag local memories by maturity: draft, private, or ready to share. Only ready memories are promoted to the shared store with source attribution.

    Why it helps: Reduces noise in shared memory and protects sensitive or unverified context.

    Linked archetype: silent-dependency-break

    Example. A Researcher agent drafts three candidate approaches to a problem and labels them draft. Only the final recommended approach, once verified, is promoted to the shared store as a shareable fact with a source link. The earlier discarded drafts never pollute the memory that the Builder or Runtime roles query.

    Anti-pattern 2.1: Drifting context¶

    Statement: Summarisation or process-local storage loses constraints, identity, or goal information that later agents need.

    Symptoms: A restarted agent defaults to a generic identity; a long thread drifts from the original goal after repeated summarisation.

    Consequence: Decisions are made with stale or incomplete context.

    Linked archetype: drifting-context

    Example. A long-running task is summarised to fit a context window. The summary drops a constraint: "do not restart the database." The next agent in the thread proposes a recovery step that requires a database restart. The original constraint is gone, and the proposal would have caused data loss.

    Anti-pattern 2.2: Silent dependency break¶

    Statement: An agent changes shared state without notifying other agents that depend on it.

    Symptoms: One agent writes to a different identity namespace, regenerates a shared secret, or changes a config file; other agents fail hours later with no obvious cause.

    Consequence: Cascading failures, long debugging loops.

    Linked archetype: silent-dependency-break

    Example. A Runtime agent rotates a shared credential used by the Archivist to authenticate to the memory service. The rotation succeeds locally, but the Archivist is not informed. Hours later every memory lookup fails with an authentication error, and the failure is first noticed as unrelated "amnesia" in downstream tasks.

    3. Verification patterns¶

    Pattern 3.1: Gate before forward motion¶

    Statement: No fleet-approved change moves to the next phase until a Verifier signs off on the current phase.

    How it works: Each phase has written pass/fail criteria. The Verifier runs the criteria and posts a green/red verdict before the Dispatcher advances.

    Why it helps: Prevents accumulating unverified debt across phases.

    Linked archetype: tool-success-mission-failure

    Example. A migration is split into three phases: render manifests, apply to staging, apply to production. The Verifier defines that the staging phase is green only when the health endpoint reports 200 for five minutes. The Dispatcher refuses to schedule production until the Verifier posts green; the Runtime role cannot bypass the gate.

    Pattern 3.2: Calibrate evaluation metrics to the task distribution¶

    Statement: A success metric must distinguish between successful outcomes in the target workload.

    How it works: If a scoring formula collapses to zero for common successful cases, rescale or re-parameterise it.

    Why it helps: A metric that cannot distinguish success is worse than no metric.

    Linked archetype: tool-success-mission-failure

    Example. A benchmark scores agents on task completion time alone. Long but correct solutions receive the same score as failures. A tournament that compares two agents becomes meaningless because both fast and slow successes look identical to the metric. The scoring is recalibrated to reward correctness first and efficiency second.

    Anti-pattern 3.1: Tool success, mission failure¶

    Statement: Each step succeeds, but the composed workflow fails because there is no end-to-end gate.

    Symptoms: Template renders, build passes, tests pass, but the deployed system is broken; benchmark tasks score as failed despite producing correct outputs.

    Consequence: False confidence; regressions discovered late.

    Linked archetype: tool-success-mission-failure

    Example. A deploy pipeline reports every step green: the template rendered, the container built, and the unit tests passed. But the rendered manifest dropped a required environment variable. The deployed service starts and then immediately crashes. No end-to-end health gate existed, so the failure surfaces in production instead of at the pipeline.

    Anti-pattern 3.2: Surface-level drift¶

    Statement: Docs, scripts, and code examples diverge from the actual system shape.

    Symptoms: New adopters follow a tutorial that uses flat config keys while the loader expects nested keys; tool names differ between README and runtime; quickstart emits URLs for a component that was removed.

    Consequence: First-try failures, support load, erosion of trust.

    Linked archetype: surface-level-drift

    Example. A quickstart guide tells a new operator to run a setup command using one binary name, but the binary has been renamed. The tutorial also emits a URL for a reverse proxy that was removed two releases ago. Each document is internally consistent; together they are impossible to follow on the first try.

    4. Build / deploy / runtime patterns¶

    Pattern 4.1: Coupled urgent-fix bundles ship together¶

    Statement: When multiple fixes are tightly coupled, land them as one unit with a single merge/deploy/rollback plan.

    How it works: The Builder writes a merge plan; the Runtime agent writes a deploy plan; the Verifier writes a rollback plan. The fleet reviews all three before execution.

    Why it helps: Prevents partial landing where one fix is on master and the runtime is still down.

    Linked archetype: loop-without-progress

    Example. Three separate bugs each have a fix approved, but each sits on its own branch. The live system remains broken because no single plan coordinates merging, deploying, and verifying all three together. The fleet bundles them into one urgent-fix unit with a shared merge/deploy/rollback plan and lands them in one coordinated step.

    Pattern 4.2: Idempotent secret bootstrap¶

    Statement: Live secrets must be created once and preserved across upgrades; rotation must be a separate, deliberate operation.

    How it works: A bootstrap script creates missing secrets and skips existing ones. Routine upgrades do not pass secret values. A CI check verifies that every expected secret is present in the rendered manifest.

    Why it helps: Stops deploy tools from accidentally invalidating credentials.

    Linked archetype: ungoverned-secret-mutation

    Example. A bootstrap script runs on a fresh environment and creates API keys, database passwords, and signing secrets only if they do not already exist. A routine upgrade later renders the manifest without passing secret values; the CI check confirms the secrets are still referenced. No rotation happens by accident during a normal deploy.

    Pattern 4.3: Render the desired state, never suppress it¶

    Statement: In declarative deployment tools, always emit the resource manifest; use live lookups only to preserve existing values, not to gate emission.

    How it works: Declarative templates always render the desired resource. A live lookup reads current values so the upgrade is a no-op; a retention policy prevents accidental deletion.

    Why it helps: Declarative tools prune resources they no longer see. Suppressing a resource to “protect” it causes the exact failure it was meant to prevent.

    Linked archetype: tool-success-mission-failure

    Example. A deployment tool is configured to skip rendering a Secret when it already exists in the cluster, intending to protect live credentials. The tool interprets the suppressed manifest as "this Secret is no longer desired" and deletes it. The correct approach is to always render the Secret and use a live lookup to preserve its existing value.

    Anti-pattern 4.1: Ungoverned secret mutation¶

    Statement: A helper script or automation regenerates credentials that downstream consumers already hold.

    Symptoms: A render-values helper is run against an environment file; it regenerates API keys and passwords and applies them. Services start, but agents cannot authenticate.

    Consequence: Auth outage, credential drift, manual recovery.

    Linked archetype: ungoverned-secret-mutation

    Example. A Runtime agent runs a values-rendering helper against an environment file to refresh a deployment. The helper regenerates API keys, database passwords, and signing secrets, and the agent applies the new values. Services restart with the fresh credentials, but other agents and stateful consumers still hold the old ones. Within minutes the fleet reports widespread authentication failures.

    Anti-pattern 4.2: Stale hand-off¶

    Statement: One agent completes recovery or setup but leaves state bound to an environment that no longer exists.

    Symptoms: A cluster is rebuilt, but storage volumes still point to the old node identity; a kubeconfig on a workstation is stale after cluster recreation.

    Consequence: Stateful workloads cannot start; operators cannot see the cluster.

    Linked archetype: stale-hand-off

    Example. The Runtime role rebuilds a cluster after a node failure. The new cluster is healthy, but persistent volumes still reference the old node identity. When the Builder role schedules workloads, their storage cannot mount because the stateful layer is anchored to the previous world.

    5. Coordination and governance patterns¶

    Pattern 5.1: Single-owner skills and docs¶

    Statement: Every piece of agent guidance has one canonical owner and location.

    How it works: Skills carry frontmatter declaring the owner, canonical path, and related skills. Duplicates are retired or merged, not copied silently.

    Why it helps: Prevents drift between copies; agents read the same guidance.

    Linked archetype: surface-level-drift

    Example. Two profiles each contain a skill that describes how to triage a failed deploy. One copy says to re-run the bootstrap step; the other says to skip it. Depending on which profile an agent loads, it takes opposite actions. The fleet adopts a rule that every skill carries frontmatter naming a single owner role, a canonical location, and the last review date. Duplicates are retired, and agents are instructed to reject guidance that lacks the owner tag.

    Pattern 5.2: Drive forward without permission stalls¶

    Statement: The Dispatcher advances multi-phase work to the next gate, convening the fleet at milestones, and only escalates to the human on true blockers.

    How it works: At the end of each phase, the Dispatcher asks: what is blocked, who owns it, and can the fleet decide? If yes, continue. If no, escalate.

    Why it helps: Reduces human coordination load; keeps momentum.

    Linked archetype: loop-without-progress

    Example. A migration has five phases. At the end of phase two the Dispatcher convenes the Builder, Runtime, and Verifier roles. All three agree the staging gate is green and the rollback plan is current. The Dispatcher advances to phase three without waiting for a new human prompt. It only escalates when phase four surfaces an unresolved dependency.

    Pattern 5.3: Post-mission review as ritual¶

    Statement: After every significant migration or incident, the fleet convenes a review and updates skills, runbooks, or conventions.

    Why it helps: Captures lessons before they decay; turns incidents into durable capability.

    Linked archetypes: all

    Example. After a secret-rotation outage, the Runtime and Archivist roles convene a review. They update the skill that describes secret handling to require explicit opt-in and a re-seeding plan. They also add a CI check that fails if a render helper is run against a live environment file. The next similar task follows the updated guidance.

    Anti-pattern 5.1: Decisions without landing rituals¶

    Statement: The fleet approves changes but does not define who merges, deploys, verifies, and rolls back.

    Symptoms: Urgent fixes exist on branches weeks after approval while the main branch remains broken; everyone assumes someone else will land the work.

    Consequence: Approved work rots; runtime and codebases diverge.

    Linked archetype: loop-without-progress

    Example. The fleet agrees that a critical fix is needed. The Builder writes the patch, but no one is assigned to merge it. The Runtime role assumes the Builder will merge; the Builder assumes the Runtime role will deploy. Two weeks later the patch is still on a branch and the production system is still broken.
