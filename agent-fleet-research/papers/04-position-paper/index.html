    Memory-Centric Collaboration: A Position on Scaling Agent Fleets¶

    Abstract. I argue that scaling long-running multi-agent systems depends on shared persistent state at least as much as on orchestration logic alone. The dominant mental model treats agents as discrete reasoning units coordinated by a central dispatcher. I contrast this with a memory-centric model in which agents share an externalised record of identity, goals, events, and decisions. Drawing on operational experience with a six-role agent fleet, I show that the most costly failures were caused by implicit or missing memory contracts, while the most durable strategies made those contracts explicit. I outline design principles for memory-centric collaboration and respond to common objections.

    1. Introduction¶

    The prevailing way to design multi-agent systems is orchestration-centric. One agent, often called a planner or manager, decides which agent should act next. The emphasis is on routing, delegation, and conversation flow. This model works well for short tasks with clear goals and fresh context. It works less well when agents operate for weeks, recover from restarts, or inherit responsibilities started by another instance.

    I operated a fleet organised around six roles over an extended period. The failures that consumed the most time were rarely failures of reasoning. They were failures of shared state: a session defaulting to the wrong identity, a credential rotated without telling downstream agents, a long thread summarised until a constraint disappeared, a decision approved but never recorded as a landing plan. In each case, the orchestration logic was doing what it was told; the missing element was memory.

    This paper makes a simple but consequential claim: in long-running agent systems, shared persistent state is at least as important as orchestration. I call this the memory-centric view.

    2. The orchestration-centric view and its limits¶

    In the orchestration-centric view, the central problem is: given a task, which agent should do what, in what order? The solution is a dispatcher, a workflow graph, or a conversation loop. Memory, if it exists, is treated as an optional cache for convenience.

    This view has produced useful systems. It also has predictable limits:

    • Identity is implicit. A session resolver may store state in-process, so a restarted agent defaults to a generic identity and forgets the active task.

    Example. A user asks the fleet to refactor a configuration store. The conversation spans ten turns, then the agent process restarts. Because session state is in-process, the new instance defaults to a generic identity and asks the user to restate the goal. The original thread is lost.

    • Goals are mutable. A long conversation is summarised to save context, and the summary silently drops a constraint.

    Example. A twelve-turn thread comparing deployment strategies is summarised to fit the context window. The summary drops the constraint that the service must stay available. The next agent proposes a strategy that requires a brief outage. The constraint was not rejected; it was compressed away.

    • Changes are invisible. One agent writes to a different namespace or rotates a secret; other agents fail hours later with no obvious cause.

    Example. A Runtime role rotates a credential used by the Archivist to write to the shared memory service. The rotation succeeds locally, but no event is broadcast. Hours later, memory lookups fail with authentication errors, and the symptom is first diagnosed as amnesia in unrelated tasks.

    • Decisions evaporate. The fleet agrees on a fix but records no merge, deploy, verification, or rollback plan, so approved work rots.

    Example. The fleet agrees a critical fix is needed. The Builder writes the patch, but no landing plan is recorded. The Runtime role assumes the Builder will merge; the Builder assumes the Runtime role will deploy. Two weeks later the patch is still on a branch and production is still broken.

    These are all failures of memory in the broad sense: the system cannot remember who it is, what it decided, what changed, or what it still owes.

    3. What memory-centric collaboration means¶

    A memory-centric model treats shared state as a first-class design primitive, not as a caching layer. The shared record contains at least:

    • Identity. Who is acting and on whose behalf.
    • Goal artefacts. Immutable or versioned records of the original goal and current phase.
    • Event log. A chronological, append-only record of changes made by any role.
    • Contracts. Ownership, deliverables, verification gates, and landing plans.
    • Local state. Per-agent drafts, preferences, and transient scaffolding that is not shared until promoted.

    In this model, orchestration still matters, but it operates on top of the shared record. The Dispatcher’s job is not merely to route; it is to keep the shared record consistent with reality.

    Example. The Dispatcher advances a migration only after the Verifier posts a green verdict in the shared record. The verdict is not a private message; it is an artefact that other roles can query. If the Dispatcher restarts, it resumes from the shared record rather than from conversation context.

    4. Evidence from the field study¶

    The companion field study catalogue contains several patterns that illustrate the memory-centric view:

    • Externalise session state. Session affinity survives restarts because session metadata lives outside any single process.
    • Explicit identity in shared tools. Memory writes require caller and subject identity; defaults are forbidden.
    • Local-first, shared-when-useful. Only stable, source-attributed facts are promoted to the shared store.
    • Single-owner skills and docs. Guidance has a canonical owner and location, preventing conflicting copies from becoming the de facto memory.
    • Gate before forward motion. A Verifier posts a green/red verdict that becomes part of the shared record before the Dispatcher advances.
    • Render the desired state. Declarative tools always emit the manifest, so the shared record matches the desired state.

    Each of these is a memory contract. Each prevents a failure that orchestration alone could not catch.

    5. Design principles¶

    From the field-study patterns I derive the following principles for memory-centric collaboration:

    1. Externalise before scaling. Do not promise horizontal scale or recovery until session state, identity, and goal artefacts live outside individual processes.
    2. Make identity explicit. Every write to shared memory must carry caller and subject identity. Reject ambiguous calls rather than silently default.
    3. Version or freeze goals. The original goal and current phase should be recoverable even after summarisation or restart.
    4. Append events, not just state. A change log makes causality visible and debugging tractable.
    5. Separate local from shared. Transient scaffolding belongs to the agent; cross-agent facts belong to the shared store.
    6. Treat decisions as artefacts. Every approved change should produce a merge plan, deploy plan, verification gate, and rollback plan in the shared record.
    7. Make contracts observable. The fleet should be able to query who owns what, what is blocked, and what has been verified.

    Example. A secret rotation is approved. The shared record contains: the Runtime role owns rotation, the Archivist owns re-seeding, the Verifier owns the post-rotation health gate, and a rollback plan is attached. Any role can query the record to see the current phase and who is blocked.

    6. Counterarguments and responses¶

    “Centralised memory creates a single point of failure.”

    It does, which is why local-first, in-memory fallbacks, and event logs are part of the model. The goal is not to replace local state but to define what must be shared and what can be recovered.

    Example. If the shared memory service is unavailable, agents fall back to local state and append events to a local log. When the service returns, the local events are replayed into the shared record. The system degrades gracefully rather than forgets.

    “Explicit contracts slow the fleet down.”

    They slow some tasks down, particularly when the fleet is learning a new convention. They speed up recovery, debugging, and coordination in steady state. The net effect depends on task duration and failure cost; in long-running operations, the balance favours contracts.

    “This is just good software engineering, not a new AI insight.”

    That is partly true. Many of these ideas come from SRE, databases, and distributed systems. The new part is applying them to agents whose context is bounded, whose identities are easy to lose, and whose actions are generated rather than programmed. The orchestration-centric view dominates current agent tooling; the memory-centric view is an alternative design centre.

    7. Implications for tool builders¶

    A memory-centric stance has implications for frameworks and platforms:

    • Provide shared memory as a primitive, not an afterthought.
    • Require identity on every read and write; do not make generic buckets the default.
    • Expose an append-only event log that agents and operators can query.
    • Support promotion workflows from local to shared state.
    • Treat skills, runbooks, and examples as shared memory with owners and version history.

    Example. A shared-memory tool should reject a write unless both caller identity and subject identity are supplied. A generic default bucket should be opt-in, not the default, because it hides misrouting in multi-agent systems.

    8. Conclusion¶

    I argued that scaling long-running multi-agent systems depends on shared persistent state at least as much as on orchestration. The memory-centric model makes identity, goals, events, and contracts first-class, externalised primitives. The patterns observed in a six-role operations fleet support this view: the most durable strategies all make memory contracts explicit, while the most damaging failures occurred when those contracts were implicit or missing. I offer this position as a design heuristic for practitioners and as a research question for the agent-systems community.
