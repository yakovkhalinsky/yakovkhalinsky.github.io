    Toward an Evaluation Protocol for Persistent Multi-Agent Workflows¶

    Abstract. Existing benchmarks for LLM agents focus on single-turn reasoning, tool use, or short cooperative tasks. They do not capture the coordination behaviours that matter in long-running operations fleets: role fidelity, memory correctness, identity preservation, and recovery after restart or scale. I propose a lightweight evaluation protocol for persistent multi-agent workflows. The protocol defines tasks that exercise the patterns observed in a companion field study, metrics for each task, and a minimal procedure for running and scoring them. The goal is to make multi-agent coordination measurable, comparable, and improvable.

    1. Introduction¶

    For single models, the research community has broad coverage of reasoning, coding, and knowledge. For multi-agent systems, promising frameworks exist, but rigorous evaluations are scarcer.

    I operated a six-role operations fleet over an extended period. The failures that mattered most were not solved by a better single model; they were solved by clearer contracts between roles. This paper asks how those contracts can be evaluated.

    2. Related work¶

    • SWE-bench evaluates whether a model can resolve real-world GitHub issues end to end. It is strong on software-engineering outcomes but does not test multi-agent coordination or persistence.
    • AgentBench evaluates LLMs as agents across multiple environments. It measures task success but rarely role fidelity or memory correctness across sessions.
    • MT-bench and chatbot arena focus on response quality. They do not evaluate whether an agent fleet remembers, owns, or lands work.

    The gap is not a lack of tasks but a lack of tasks designed for long-running, multi-agent, memory-dependent workflows.

    3. Evaluation requirements¶

    From operational experience I derive five requirements for an evaluation protocol:

    1. Role fidelity. Does the right role act on the right task? Can a role decline work outside its domain?
    2. Memory correctness. After a restart or context switch, does the agent recover the original goal, identity, and constraints?
    3. Identity preservation. Can the fleet distinguish agents and roles in shared memory, or do writes collide?
    4. Recovery after restart/scale. Can the system resume a long-running workflow after process replacement or horizontal scaling?
    5. End-to-end outcome correctness. Does the composed workflow produce a correct result, not just correct intermediate steps?

    Example. A fleet is asked to refactor a configuration store. Role fidelity means the Builder, not the Runtime, owns the code change. Memory correctness means that after a restart the new agent instance still knows the original goal was a refactor, not a deletion. End-to-end correctness means the store still works after the change.

    4. Proposed tasks¶

    Each task is designed to exercise one or more requirements. They are stylised and anonymised; they do not depend on any specific system.

    Task 1: Role fidelity under ambiguity¶

    A Dispatcher receives a vague request: "something is wrong with the service." The correct response is to route to the Runtime role for health diagnosis, not to the Builder for code changes. The task is scored by whether the first routed role is appropriate and whether the specialist can decline a misrouted task.

    Example. The Dispatcher observes a slow endpoint and assumes a recent configuration change is at fault. It sends the Builder a request to revert the change. The correct behaviour is for the Builder to decline and escalate to the Runtime role, because the symptom points to infrastructure, not code.

    Task 2: Verifiable dispatch¶

    A Dispatcher is given a task and must produce a dispatch message. A correct dispatch includes a concrete deliverable and a verification criterion. The task is scored by whether a downstream Verifier can determine green/red from the dispatch alone.

    Example. A correct dispatch might read: "Update the health-check endpoint so the liveness probe returns HTTP 200 within 2 seconds; mark complete only when the Verifier reports green on staging." An incorrect dispatch reads: "Look into the health check."

    Task 3: Memory recovery after restart¶

    A conversation is interrupted by a simulated process restart. The agent must resume from externalised session state and continue the original task without asking the user to restate the goal. The task is scored by goal recall and identity consistency.

    Example. Mid-refactor, the agent process restarts. The resumed instance should retrieve the original goal and current phase from the shared memory service and continue, rather than ask, "What would you like me to do?"

    Task 4: Identity isolation in shared memory¶

    Two agents write to a shared memory tool. One omits identity fields. The evaluation checks whether the memory tool rejects the write or whether the omission causes silent misrouting. The task is scored by correctness of the tool’s validation contract.

    Example. Agent A writes a memory with explicit observer and subject identity. Agent B writes a memory with the subject identity omitted. The shared-memory tool should reject Agent B’s write, not silently place it in a generic bucket where other agents will miss it.

    Task 5: Silent dependency detection¶

    An agent changes a shared config value. A downstream agent later fails. The evaluation checks whether the fleet can trace the failure back to the change through an event log or ownership record. The task is scored by causal tracing accuracy.

    Example. A Runtime role changes a shared credential. A Researcher role fails hours later when querying the memory service. The fleet must be able to follow the event log from the Researcher failure back to the credential change and identify the Runtime role as the owner of that change.

    Task 6: End-to-end landing¶

    A change is approved in conversation. The fleet must produce a merge plan, deploy plan, verification gate, and rollback plan in the shared record. The task is scored by completeness of the landing ritual and by whether the change actually reaches the simulated target environment.

    Example. The fleet approves adding a new health check. The Dispatcher must ensure the Builder produces a patch, the Runtime role produces a deploy plan, the Verifier defines a green/red gate, and the Runtime role produces a rollback plan. The task is incomplete if any of these artefacts is missing.

    Task 7: Metric calibration¶

    A benchmark scoring function is provided that collapses successful outcomes to the same score. The agent or operator must recalibrate the function so that faster and slower successes are distinguishable. The task is scored by whether the recalibrated metric reflects task quality.

    Example. The initial scoring function gives a slow but correct agent the same score as an agent that produced no output. A correct recalibration rewards correctness first and adds a secondary efficiency score that does not erase the correctness distinction.

    5. Metrics¶

    For each task I propose the following metrics:

    • Outcome accuracy. Did the workflow reach the correct end state?
    • Role accuracy. Was the task routed to the correct role?
    • Memory fidelity. After restart, how much of the original goal and identity was preserved?
    • Contract coverage. How many required artefacts (deliverable, verification, landing plan, rollback plan) were produced?
    • Causal traceability. How many steps were needed to trace a failure to its source?
    • Metric informativeness. Does the scoring function distinguish successful outcomes by task quality rather than collapsing them?

    Metrics should be reported per task and, where possible, per pattern. A single aggregate score is less useful than a profile across the requirements.

    6. Procedure¶

    A minimal run of the protocol would follow these steps:

    1. Select tasks. Choose the tasks that match the target fleet’s responsibilities.
    2. Provide a shared environment. Give the agents a memory service, event log, and tool sandbox.
    3. Inject incidents. Seed each task with the failure condition it is designed to exercise.
    4. Run the agents. Allow the fleet to coordinate without human prompting except for true blockers.
    5. Score. Apply the metrics above, preferably with an automated Verifier role or a deterministic oracle.
    6. Report a profile. Report scores per task and per requirement, not just a single number.

    Example. A pilot run might report: role fidelity 0.85, memory recovery 0.40, identity isolation 0.70, end-to-end landing 0.30, metric informativeness 0.90. The profile immediately shows that memory and landing are the weak points, even if the fleet scores well on routing and metrics.

    7. Pilot expectations¶

    I expect a pilot run to reveal that current multi-agent systems struggle most with memory correctness and end-to-end landing. Single-turn tool use is likely to score well; role fidelity under ambiguity and silent-dependency tracing are likely to score poorly. The value of the protocol is to make those weaknesses visible and comparable across systems.

    8. Limitations and future work¶

    This protocol is a proposal, not an established benchmark. The tasks are stylised and may need adaptation to real systems. Future work includes building reference implementations, collecting human baselines, and agreeing on common task definitions. I hope the companion field study and patterns catalogue provide the empirical grounding for that work.

    9. Conclusion¶

    I proposed an evaluation protocol for persistent multi-agent workflows. The protocol focuses on role fidelity, memory correctness, identity preservation, recovery, and end-to-end outcomes. By designing tasks around the patterns observed in a long-running operations fleet, the protocol aims to measure the coordination behaviours that matter most in practice. I offer it to the evaluation community as a starting point for making agent fleets comparable and improvable.
